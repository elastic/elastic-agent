# Add Azure Auth Extension to EDOT Collector Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Register `azureauthextension` (type: `azure_auth`) from OTel Contrib into the EDOT Collector component set, mirroring the pattern used for PR #12919 (Azure Monitor receiver).

**Architecture:** The EDOT component set lives in `internal/edot/` (a separate Go sub-module). Adding a component means: importing its factory in `default.go`, adding the Go module dependency, updating the README table, updating test configs, and adding a changelog fragment.

**Tech Stack:** Go, OpenTelemetry Collector Contrib v0.152.0, `mage` build tool, `elastic-agent-changelog-tool`

## Task 1: Add `azure_auth` to the Components-Output Golden File (TDD Red Phase)

**Files:**
- Modify: `internal/edot/cmd/testdata/otel/components-output.yml`

The `TestComponentsCommand` test in `internal/edot/cmd/command_components_nofips_test.go` checks that every extension listed in this golden file appears in the actual `otel components` output. Adding `azure_auth` here first makes the test fail until the factory is registered.

- [ ] **Step 1: Add `azure_auth` to the extensions list**

Open `internal/edot/cmd/testdata/otel/components-output.yml`. Add the entry alphabetically before `cgroupruntime`:

```yaml
extensions:
  - name: azure_auth
    stability:
      extension: Alpha
  - name: cgroupruntime
    stability:
      extension: Alpha
  - name: file_storage
    stability:
      extension: Beta
  # ... rest unchanged
```

The stability value (`Alpha`) comes from the extension's `metadata.yaml`:
```yaml
status:
  stability:
    alpha: [extension]
```

## Task 2: Add `azure_auth` to the Collector Startup Test Config (TDD Red Phase)

**Files:**
- Modify: `internal/pkg/otel/testdata/all-components.yml`

The `TestStartCollector` test in `internal/edot/otelcol/run_nofips_test.go` starts a real collector with this config. Adding `azure_auth` here ensures the extension is exercised at startup.

The minimal valid config requires exactly one of `managed_identity`, `workload_identity`, `service_principal`, or `use_default: true`. System-managed identity (no `client_id`) is the simplest and has a no-op `Validate()`.

- [ ] **Step 1: Add `azure_auth` to the `extensions:` block**

In `internal/pkg/otel/testdata/all-components.yml`, add to the `extensions:` section (after `awslogs_encoding/elbaccess:` and before `azure_encoding:`):

```yaml
extensions:
  health_check:
  memory_limiter:
    check_interval: 5s
    limit_mib: 100
  headers_setter:
    headers:
      - action: insert
        key: X-Scope-OrgID
        from_context: tenant_id
        default_value: Org-ID
  awslogs_encoding/elbaccess:
    format: elbaccess
  azure_auth:
    managed_identity:
  azure_encoding:
```

- [ ] **Step 2: Add `azure_auth` to `service.extensions`**

In the `service:` → `extensions:` list, add `- azure_auth` (keep alphabetical order with azure entries):

```yaml
service:
  extensions:
    - memory_limiter
    - health_check
    - headers_setter
    - awslogs_encoding/elbaccess
    - azure_auth
    - azure_encoding
```
